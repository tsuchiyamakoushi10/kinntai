## 1. プロジェクト概要

- **名称**: kinntai
- **目的**: 介護事業所向けの勤怠管理 Web アプリ
- **初期想定規模**: 5拠点・約55名
- **方針**: 「現場スタッフが迷わず使えること」を最優先する。エンジニア向けの設計より、介護現場の業務フローに沿うことを優先する。

## 2. ドメイン用語集

介護現場・労務管理特有の用語が頻出します。コード・UI・コミットメッセージで揺らがないよう、以下の表記に統一してください。

### シフト種別

施設・サービス区分ごとにパターン名が分かれる。詳細は [docs/shift-patterns.md](docs/shift-patterns.md) を参照。

| 表記                                       | 意味                                                                      |
| ------------------------------------------ | ------------------------------------------------------------------------- |
| **早**                                     | 早朝開始のシフト（例 6:30 - 15:30）                                       |
| **日勤 / デ日 / ショ日**                   | 日中の通常シフト（例 8:15 - 17:15）。施設ごとに code を分ける             |
| **遅**                                     | 夕方からのシフト（例 10:00 - 19:00）                                      |
| **夜入**                                   | 夜勤前半（例 16:30 - 24:00）。当日のシフトとして貼る                      |
| **夜明**                                   | 夜勤後半（例 0:00 - 8:30）。翌日のシフトとして貼る                        |
| **半日A〜F / 半日午後**                    | パートの定常短時間勤務（半休ではない）                                    |
| **デ短A〜E / ショ短A / 梨1〜5 / 厨房A〜C** | 拠点固有の短時間〜フル勤務パターン                                        |
| **公休**                                   | 法定休日                                                                  |
| **有休**                                   | 年次有給休暇                                                              |
| **欠勤**                                   | 無給の休み                                                                |
| **希望休**                                 | 従業員が事前申請する休み希望（管理者承認で確定）                          |
| **有/日, 日/有**                           | 1 日の中で午前 / 午後の片方に有休、もう片方に勤務を割り当てる複合パターン |

### 雇用形態

| 表記         | 意味                                 |
| ------------ | ------------------------------------ |
| **正社員**   | 月給制、フルタイム                   |
| **契約社員** | 期間定めあり、フルタイムまたは短時間 |
| **パート**   | 時給制、短時間                       |

### その他

- **所定労働時間**: 雇用契約で定めた1日あたりの労働時間
- **法定労働時間**: 労基法で定められた1日8時間・週40時間
- **割増賃金**: 時間外・深夜・休日に対して支払う割増分（深夜は 22:00 - 翌5:00）
- **基準日 / 雇い入れ日**: 有給付与の起点となる入社日
- **処遇改善加算 / 特定処遇改善加算**: 介護報酬の加算制度（将来対応）
- **キャリアアップ助成金**: 非正規雇用者の処遇改善で支給される助成金（将来対応、必要情報は要件書参照）
- **年5日取得義務**: 年10日以上の有給を持つ従業員に対し、年5日の取得を使用者側が確保する労基法上の義務（2019年4月施行）
- **労働条件通知書**: 雇い入れ時に労基法で交付が義務付けられている文書。雇用契約書と兼用可
- **職種**: 業務上の役割区分（介護職員 / 看護職員 / 生活相談員 / ケアマネ / 事務 など）。雇用形態とは別軸
- **社労士**: 社会保険労務士。労務関連書類のチェック・行政手続代行を依頼する外部専門家

## 3. 設計原則

### 3.1 UI 原則

- **打刻画面は1タップで完結すること**。確認ダイアログは原則出さない（誤打刻は管理者修正で対応）。
- **ボタンは大きく、コントラスト強め**に。共有タブレットを高齢の現場職員が使う前提。
- **専門用語をUIに出さない**。「所定労働時間」より「契約上の1日の勤務時間」のような表現を優先する。
- **画面遷移は浅く**。従業員側は2階層以内を目標。

### 3.2 ロジック原則

- **日付の扱い**は必ず `Asia/Tokyo` で統一。サーバー時刻が UTC でも、業務日付（特に夜勤跨ぎ）の判定はJST基準で行う。
- **夜勤の勤怠日付**は「出勤した日」で集計する。明けの日は「夜勤の継続」として記録する。
- **有給付与日数**は労基法準拠（雇い入れ後6か月で10日付与、以後年次で増加、所定労働日数に応じて比例付与）。実装は `src/lib/leave/` にまとめ、テストを必ず書くこと。
- **打刻のタイムスタンプは秒単位で保存**するが、UI表示は分単位。

### 3.3 データ原則

- 拠点（office）は将来 4 つ以上に増える可能性がある。`office_id` を主要テーブルに必ず持たせる。
- 個人情報（氏名、生年月日、雇用契約情報）は将来的に暗号化または列レベルアクセス制御を検討する余地を残す。MVP では DB レベルの暗号化はしないが、ログに PII を出力しないこと。
- マルチテナント化（複数法人）は MVP では非対応だが、`tenant_id` を後付けできるよう、テーブル設計時にコメントを残す。

## 4. コーディング規約

- **言語**: TypeScript（`strict: true`）
- **命名**:
  - DB カラム / テーブル: `snake_case`
  - TypeScript 変数・関数: `camelCase`
  - React コンポーネント / 型: `PascalCase`
  - ドメイン用語は英訳せず、必要なら型名にローマ字を使う（例: `ShiftKubun` ではなく `ShiftType` と `enum` 値で `EARLY` `DAY` `LATE` `NIGHT` `AFTER_NIGHT` などを使う）
- **コミットメッセージ**: 簡潔に、変更の "Why" を書く。Conventional Commits の prefix（feat / fix / docs / refactor / test / chore）を推奨。
- **PR**: 1 PR 1 トピック。レビュアが30分以内で読み切れる粒度に分割する。

## 5. やってはいけないこと

- 本番 DB / ステージング DB に対して、ユーザー確認なしに `prisma migrate deploy` や `db push` を実行しない。
- 個人情報（氏名・電話番号・住所）をログ・エラートラッキング・テストフィクスチャに **そのまま** 含めない。テストデータは架空の名前を使う。
- 「処遇改善加算」「キャリアアップ助成金」の計算ロジックを、根拠資料なしに推測で実装しない。要件書の対応箇所にTODOを残し、確認を求めること。
- 労働時間・有給日数に関するロジックを、テストなしでマージしない。
- `--no-verify` での commit や、`force push` を勝手に実行しない。
- 設計ドキュメントと矛盾する実装を勝手に進めない。差異が出たらドキュメントを先に更新するか、ユーザーに確認する。

## 6. 作業手順テンプレート

新機能を追加するとき:

1. `docs/requirements.md` に該当機能があるか確認。なければ追記してから実装。
2. `docs/screen-list.md` に画面追加が必要か確認。
3. DB 変更が必要なら `docs/database-design.md` を更新してから `prisma/schema.prisma` を編集。
4. ドメインロジックは `src/lib/` に切り出し、ユニットテストを書く。
5. UI を実装。共有タブレット幅（横向き 1024×768 程度）と個人スマホ幅（375×667）の両方で目視確認。
6. 変更点を README または該当ドキュメントへ反映。

## 7. メモ
