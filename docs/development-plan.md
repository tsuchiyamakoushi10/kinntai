# 開発計画

> **2026-05-20 方針変更**: Phase 1 のスコープを「人事 / 契約 / 書類 / 退職 / シフト制約 / 自動シフト」に絞り直した。打刻機能・勤怠承認・有給は Phase 2 / 3 に移動。打刻はハードウェア整備と現場合意を待つため、社内検証可能な人事系を先行させる。

## 1. 全体方針

- **管理者だけで触れる範囲（人事 / シフト）を先に完成させ、社内検証を回す**。打刻は現場合意が整ってから着手する。
- フェーズごとに「完了の定義」を決め、達成したら次フェーズへ移る。
- 法令関連ロジック（労働時間、有給、年 5 日取得）と、現場が毎日触る打刻 / 勤務表編集を、それぞれ着手フェーズで重点的に品質確保する。
- 個人情報を扱う書類管理は MVP の段階で「セキュリティ万全」を前提に設計する（暗号化保存、署名 URL、アクセスログ）。

---

## 2. フェーズ全体像

| フェーズ | 目的                                                     | 主要ゴール                                        | 想定期間（目安） |
| -------- | -------------------------------------------------------- | ------------------------------------------------- | ---------------- |
| Phase 0  | 設計確定                                                 | 本ドキュメント群の合意                            | 1〜2 週          |
| Phase 1  | MVP: 人事 / 契約 / 書類 / 退職 / シフト制約 / 自動シフト | 管理者だけで「人事 + シフト作成」が完結できる状態 | 8〜12 週         |
| Phase 2  | 打刻 / 勤怠閲覧 / 承認                                   | 共有タブレット + 個人スマホで打刻運用が始まる     | 4〜6 週          |
| Phase 3  | 有給 / 申請系 / 出力系                                   | 年 5 日取得チェック、修正申請、CSV 出力           | 4〜6 週          |
| Future   | 契約書 PDF / 助成金 / IC 打刻                            | 業務拡張                                          | 都度判断         |

> 期間は規模感の目安。週あたりの稼働量と検証頻度で前後する。

## 4. Phase 1 — MVP 実装（人事・契約・書類・退職・シフト制約・自動シフト）

### スコープ（[requirements.md §4.1](requirements.md#41-mvpフェーズ1) 準拠）

認証、拠点管理、社員情報、雇用契約、書類、研修、退職処理 / 退職者リスト、シフトパターン、シフト制約、シフト希望、月次シフト自動作成、月次勤務表編集（手動）。

> 認証・拠点管理は完了済（〜2026-05-19）。下記サブフェーズは社員管理スライスの拡張から再スタートする。

### サブフェーズ

#### 1-A. 基盤構築 — 完了

- Next.js プロジェクト初期化、Tailwind、Auth.js、Prisma、Lint/Format
- DB スキーマ作成（offices, users, employees, qualifications, shift*patterns, shifts, attendance_records, break_records, paid_leave*\*）
- シード（5 拠点 + 架空の従業員 55 名 + 約 30 種のシフトパターン）
- CI（型チェック、テスト、Lint）

#### 1-B. 認証・拠点管理 — 完了

- S-C-01 ログイン / S-C-02-03 パスワードリセット
- S-A-02 拠点設定
- 管理者ロール / 従業員ロールの権限ガード

#### 1-C. 社員情報拡張 + 退職処理改修 — 完了

- `employees` への `employment_status` enum / `retirement_reason` カラム追加（migration）
- S-A-04 従業員詳細をタブ構成に再設計（基本 / 契約 / 書類 / 研修 / シフト）
- S-A-05 新規 / 編集 を拡張カラム対応
- S-A-06 退職処理を「`employment_status=retired` に遷移 + 退職理由」に改修
- S-A-20 退職者一覧を新規実装

#### 1-D. 雇用契約管理 — 完了

- `employment_contracts` 追加（migration + Prisma）
- S-A-21 雇用契約 新規 / 編集（更新有無、更新上限、保険、退職金、助成金フラグ）
- S-A-04 詳細画面に「契約タブ」(履歴一覧 + 現契約サマリ) を追加
- 退職金通算判定ロジック（`src/lib/contract/retirement-allowance.ts`）+ テスト

#### 1-E. 書類管理 — 完了

- `employee_documents` / `document_access_logs` 追加（migration）
- ストレージ抽象化（`src/lib/storage/`）: ローカル FS 実装 + HMAC 署名トークン（Supabase Storage は将来差し替え可能）
- S-A-22 書類タブ（従業員詳細）: アップロード / 一覧 / 署名 URL ダウンロード / 削除 + 監査ログ
- 期限切迫アラート（S-A-01 ダッシュボードに 30 日以内 / 期限切れを集約）
- セキュリティテスト: Playwright E2E で署名検証 / token 改ざん / token 不足 / ダウンロードログを確認

#### 1-F. 研修管理 — 完了

- `training_records` 追加（migration + Prisma リレーション）
- S-A-23 研修タブ: 一覧 / 追加 / 編集 / 削除 + 費用合計 / 本人負担合計の表示
- 修了証ファイルは書類アップロード時に `training_record_id` で紐付け可能（他従業員の研修記録は弾く）

#### 1-G. シフト制約 + シフト希望 — 完了

- `shift_constraints` / `shift_preferences` 追加（migration + Prisma リレーション）
- S-A-24 個人別シフト制約 編集（従業員詳細「制約・希望」タブ）
- S-A-25 シフト希望管理（月別/拠点/状態フィルタ + accept/reject/戻す）
- S-E-10 シフト希望入力（従業員側スマホ画面）
- 年収 130 万円見込み計算ロジック (`src/lib/shift/income-projection.ts`) + 25 件のユニットテスト

#### 1-H. 月次シフト自動作成（2〜3 週）

> 詳細設計は [docs/auto-shift-design.md](auto-shift-design.md) に集約。論点 A〜G は 2026-05-21 に確定済。

サブステップ:

- **1-H-1 基盤**: Prisma `office_shift_quotas` / `shift_generation_runs` を追加、`shifts.generation_run_id` (nullable) を追加。migration + シード（5 拠点分の quota 雛形）。祝日マスター `src/lib/calendar/holidays.ts` を 2024–2030 年分で整備。
- **1-H-2 S-A-27 拠点シフト枠 設定**: 拠点 × 日種 (`weekday` / `saturday` / `sunday_holiday`) × シフトパターンのマトリクス UI。休み系パターンは除外、拠点固有 + 共通パターンを並べる。
- **1-H-3 自動作成アルゴリズム**: `src/lib/shift/auto-generator/` に純粋関数で実装。`algorithm_version = greedy-v1`。決定論性 (`seed`)、公休も自動配置、`updated_by` ベースの手動編集保護を含む（[auto-shift-design §4](auto-shift-design.md#4-ドメインロジック)）。ユニットテストで [auto-shift-design §8](auto-shift-design.md#8-テスト方針) のシナリオを網羅。
- **1-H-4 S-A-26 自動作成 UI**: 「dry-run → 下書き保存 → S-A-08 で微調整 → 確定」の 3 段フロー。警告は重大度別バッジ表示。
- **1-H-5 S-A-08 拡張**: 自動由来セルの識別表示、未確定 run のバナー、確定取り消し導線。

### Phase 1 完了の定義

- 全 Phase 1 画面が動作し、シード環境で 1 か月分のシフト自動作成 → 手動調整 → 確定 まで通る
- 退職金通算 / 130 万円見込みのロジックがテストで網羅されている
- 書類のアクセス制御（他従業員の書類は不可、署名 URL は短期失効）が動作確認できる

## 8. リスクと対策

| リスク                                | 影響                       | 対策                                                                                                                               |
| ------------------------------------- | -------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| 書類ファイル漏洩                      | 重大インシデント、法令違反 | オブジェクトストレージ + サーバ側暗号化、署名 URL（5 分有効）、`document_access_logs`、定期的なアクセスログレビュー                |
| 雇用契約 / 退職金判定の誤り           | 労務紛争リスク             | `src/lib/contract/` にロジックを隔離、テスト網羅、`retirement_allowance_eligible` で手動上書き可能にして判定保留を許容             |
| シフト自動作成が現場感覚と乖離        | 自動結果が使われない       | 自動配置はあくまで下書き。S-A-08 での手動調整 + S-A-26 で警告 / 充足率を可視化。`algorithm_version` で結果差異を遡及検証可能にする |
| 130 万円見込みの誤算で扶養超過        | 従業員の社会保険トラブル   | `src/lib/shift/income-projection.ts` のテスト網羅、80% / 100% の段階アラート、Phase 2 で実績ベースに切替                           |
| 夜勤跨ぎの日付処理ミス（Phase 2）     | 勤怠改ざん / 給与計算ミス  | `attendance_records.work_date` の規約を [`database-design.md §6`](database-design.md#6-設計上の留意点) で明確化、テストで網羅      |
| 有給付与日数ロジックのバグ（Phase 3） | 法令違反、未払い問題       | 労基法準拠の付与日数を表で持ち、Property-based test で全パターン検証                                                               |
| 共有タブレットでの誤打刻（Phase 2）   | 信頼性低下                 | 暗証番号で本人確認、打刻直後 3 秒間は同種打刻を抑止、直近打刻の常時表示                                                            |
| 個人情報漏洩                          | 重大インシデント           | PII をログから除外、HTTPS 強制、定期的な依存性更新、退職者アカウント無効化フロー                                                   |
| 現場で使われない                      | プロジェクト失敗           | Phase 1 完了時点で社内デモ → Phase 2 で先行運用しフィードバックを反映、ボタンの大きさ / 文言を実機で検証                           |
| 法令変更                              | ロジック修正が必要         | 割増率・有給日数などは定数または DB に外出し、根拠資料を docs に蓄積                                                               |
| 年 5 日取得義務違反（Phase 3）        | 法令違反、行政指導リスク   | S-A-17 で取得不足者を月次アラート、付与日からの残期間が 3 か月を切ったら警告を強める                                               |

## 9. 成功指標（KPI 案）

### Phase 1 KPI（人事 / シフト）

| 指標                                       | 目標                            |
| ------------------------------------------ | ------------------------------- |
| シフト自動作成の制約満足率                 | 95% 以上（残り 5% は警告表示）  |
| 月次シフト作成時間（管理者 1 人あたり）    | 紙 / Excel 運用比 1/2 以下      |
| 書類アップロード〜閲覧までの平均クリック数 | 3 クリック以内                  |
| 社員情報の重複入力箇所                     | 0（契約スナップショットを除く） |

### Phase 2 / 3 KPI（打刻 / 集計）

| 指標                                 | 目標              |
| ------------------------------------ | ----------------- |
| 1 打刻あたりの操作時間（タブレット） | 5 秒以内          |
| 打刻失敗率                           | 0.5% 未満         |
| 月締め作業時間（管理者 1 人あたり）  | 紙運用比 1/3 以下 |
| 修正申請率（打刻に対する）           | 5% 未満           |

Phase 1 KPI は社内検証フェーズで計測開始。Phase 2 / 3 の KPI は打刻運用開始後に計測し、改善判断の材料にする。
