# 開発計画

## 1. 全体方針

- **MVP を最短で現場に届け、運用しながら磨く**。設計の完璧さより、現場フィードバックの取りやすさを優先する。
- フェーズごとに「完了の定義」を決め、達成したら次フェーズへ移る。
- 法令関連ロジック（労働時間、有給）と、現場が毎日触る打刻 / 勤務表編集を最優先で品質確保する。

---

## 2. フェーズ全体像

| フェーズ | 目的                   | 主要ゴール                   | 想定期間（目安） |
| -------- | ---------------------- | ---------------------------- | ---------------- |
| Phase 0  | 設計確定               | 本ドキュメント群の合意       | 1〜2 週          |
| Phase 1  | MVP 実装               | 1 事業所で社内検証できる状態 | 6〜10 週         |
| Phase 2  | 試験運用               | 5 拠点 55 名で本番運用開始   | 4〜6 週          |
| Phase 3  | 申請系・出力系         | 修正申請 / 希望休 / CSV出力  | 4〜6 週          |
| Future   | 契約書・助成金・IC打刻 | 業務拡張                     | 都度判断         |

> 期間は規模感の目安。週あたりの稼働量と検証頻度で前後する。

## 4. Phase 1 — MVP 実装

### スコープ（[requirements.md §4.1](requirements.md#41-mvpフェーズ1) 準拠）

認証、打刻、勤怠閲覧、月別集計、勤務表、シフトパターン、従業員管理、有給管理。

### サブフェーズ

#### 1-A. 基盤構築（1〜2 週）

- Next.js プロジェクト初期化、Tailwind、Auth.js、Prisma、Lint/Format
- DB スキーマ作成（offices, users, employees, qualifications, shift_patterns）
- シード（5 拠点 + 架空の従業員 55 名 + 約 30 種のシフトパターン）
- CI（型チェック、テスト、Lint）

#### 1-B. 認証・従業員管理（1〜2 週）

- S-C-01 ログイン / S-C-02-03 パスワードリセット
- S-A-03〜S-A-06 従業員管理
- S-A-02 拠点設定
- 管理者ロール / 従業員ロールの権限ガード

#### 1-C. 打刻（2 週）

- S-T-01〜S-T-05 共有タブレットフロー
- S-E-01 ホーム + スマホ打刻
- `attendance_records` / `break_records` の整合性（夜勤跨ぎ含む）
- 打刻 API のユニットテスト + E2E（Playwright）

#### 1-D. シフトパターン / 勤務表（2 週）

- S-A-07 シフトパターン管理
- S-A-08 月次勤務表編集（キーボード操作対応、コピー機能）
- S-E-04 自分の月次シフト表示

#### 1-E. 勤怠閲覧・承認・集計（2 週）

- S-E-02 / S-E-03 月別勤怠
- S-A-09 勤怠承認
- S-A-10 月別集計（実労働 / 残業 / 深夜 / 有休）
- 集計ロジックは `src/lib/attendance/` に分離、テスト必須

#### 1-F. 有給管理（1〜2 週）

- 雇い入れ日に基づく自動付与バッチ（日次 cron）
- S-A-11 / S-E-05
- 比例付与（パート対応）、繰越、2 年失効
- **年5日取得義務チェック**（S-A-17）: 取得不足者の一覧表示

### Phase 1 完了の定義

- 全 MVP 画面が動作し、シード環境で 1 か月分の運用シミュレーションを完了
- 法令ロジック（労働時間集計、有給付与）のテストが網羅されている
- 共有タブレット（iPad 想定）と iPhone Safari で打刻が問題なく動作

## 5. Phase 2 — 試験運用

### スコープ

- 自社 / パートナー事業所の 1 拠点で先行運用 → 順次 5 拠点に展開
- バグ修正と現場 UX 調整に注力（新機能は最小限）

### 主タスク

- 本番環境構築（Vercel + Supabase 想定、または社内サーバ Docker）
- バックアップ / リストア手順の確立
- 運用マニュアル作成（管理者向け / 従業員向け、A4 1〜2 枚）
- 初期データ投入支援（既存の Excel シフト表から取り込むスクリプト）
- 監視（エラー率、レスポンスタイム、打刻失敗）
- 障害対応窓口の整備

### 完了の定義

- 5 拠点 55 名で月締めを 1 サイクル無事に完了
- 重大バグ（打刻ロスト、勤怠改ざんリスク、ログイン不可など）ゼロ

## 6. Phase 3 — 申請系・出力系

### スコープ

- 打刻修正申請（S-E-07, S-A-12）
- 希望休申請（S-E-08, S-A-13）
- 申請一覧（S-E-09）
- 月次集計 CSV / 給与ソフト連携 CSV（S-A-14）
- 拠点管理者ロールの分離（必要に応じて）
- 通知（メール、または LINE）

### 完了の定義

- 紙運用の修正申請 / 希望休が完全に廃止できる
- 給与ソフト（候補: マネーフォワード給与、freee 人事労務 など）に取り込める CSV が出力できる

## 8. リスクと対策

| リスク                     | 影響                      | 対策                                                                                                                          |
| -------------------------- | ------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| 夜勤跨ぎの日付処理ミス     | 勤怠改ざん / 給与計算ミス | `attendance_records.work_date` の規約を [`database-design.md §6`](database-design.md#6-設計上の留意点) で明確化、テストで網羅 |
| 有給付与日数ロジックのバグ | 法令違反、未払い問題      | 労基法準拠の付与日数を表で持ち、Property-based test で全パターン検証                                                          |
| 共有タブレットでの誤打刻   | 信頼性低下                | 暗証番号で本人確認、打刻直後3秒間は同種打刻を抑止、直近打刻の常時表示                                                         |
| 個人情報漏洩               | 重大インシデント          | PII をログから除外、HTTPS 強制、定期的な依存性更新、退職者アカウント無効化フロー                                              |
| 現場で使われない           | プロジェクト失敗          | Phase 2 で先行運用しフィードバックを反映、ボタンの大きさ / 文言を実機で検証                                                   |
| 法令変更                   | ロジック修正が必要        | 割増率・有給日数などは定数または DB に外出し、根拠資料を docs に蓄積                                                          |
| 年5日取得義務違反          | 法令違反、行政指導リスク  | S-A-17 で取得不足者を月次アラート、付与日からの残期間が3か月を切ったら警告を強める                                            |

## 9. 成功指標（KPI 案）

| 指標                                 | 目標              |
| ------------------------------------ | ----------------- |
| 1 打刻あたりの操作時間（タブレット） | 5 秒以内          |
| 打刻失敗率                           | 0.5% 未満         |
| 月締め作業時間（管理者 1 人あたり）  | 紙運用比 1/3 以下 |
| 修正申請率（打刻に対する）           | 5% 未満           |

これらは Phase 2 開始後に計測を始め、Phase 3 着手判断の材料にする。
