# MVP スコープ詳細

[`requirements.md`](requirements.md) の「機能要件」をフェーズ別に詳細化したもの。「含む / 含まない / 理由」を明示し、判断のブレを防ぐ。

---

## 1. スコープ判定の方針

MVP 含めるか否かは以下の基準で判断する。

| 基準                    | 内容                                                                               |
| ----------------------- | ---------------------------------------------------------------------------------- |
| **法令準拠**            | 法令違反リスクのある機能（労働時間集計、有給付与、年5日取得チェック）は MVP に含む |
| **毎日使う / 月次必須** | 打刻・シフト・勤怠承認など、運用上欠かせない機能は MVP に含む                      |
| **代替手段の有無**      | 紙・Excel で当面回せる機能は Phase 2 以降に回す（例: CSV出力、希望休申請）         |
| **影響範囲**            | 1名でも漏れたら困る機能は MVP に含む（例: 有給付与）                               |
| **UI 複雑度**           | 高度な UI が必要で MVP の検証を遅らせる機能は Phase 2 以降に回す                   |

## 2. MVP（Phase 1）— 「これがないと運用が回らない」

### 2.1 含む機能

| カテゴリ              | 機能                                                                                                                               | 必須理由                                                          |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------- |
| 認証                  | 管理者ログイン / 従業員ログイン / 共有タブレット打刻専用モード                                                                     | システム利用の前提                                                |
| 打刻                  | 出勤 / 退勤 / 休憩開始 / 休憩終了                                                                                                  | 全機能の基礎データ                                                |
| 打刻方法              | Web ブラウザ（共有タブレット + 個人スマホ）                                                                                        | 既存ハードで運用可能                                              |
| 勤怠閲覧              | 従業員: 自分の月別勤怠                                                                                                             | 透明性・本人確認                                                  |
| 勤怠閲覧              | 管理者: 全員の月別勤怠                                                                                                             | 月締めに必須                                                      |
| 勤怠承認              | 月締め時の個別 / 一括承認                                                                                                          | 給与連携の前提                                                    |
| 月別集計              | 出勤日数、実労働、所定内 / 外、深夜、休日、有休消化（画面表示）                                                                    | 月締めに必須                                                      |
| 勤務表                | 月次勤務表 (S-A-08)、シフトパターン割当、前月コピー                                                                                | 介護現場の運用必須                                                |
| シフトパターン        | 早 / 日勤(デ日/ショ日) / 遅 / 夜入 / 夜明 / 半日勤務 / 拠点固有パターン / 公休 / 有休 / 欠勤 / 希望休 / 複合（有/日・日/有）の定義 | 勤務表作成の前提。詳細は [`shift-patterns.md`](shift-patterns.md) |
| 従業員管理            | 新規登録・編集・退職処理、緊急連絡先、職種、資格情報                                                                               | 人事管理の最低限                                                  |
| 拠点管理              | 拠点の追加・編集                                                                                                                   | 5拠点運用の前提                                                   |
| 有給付与              | 雇い入れ日基準の自動付与、比例付与、繰越、2年失効                                                                                  | 法令準拠                                                          |
| 有給残数表示          | 従業員ホームへの常時表示                                                                                                           | 取得促進                                                          |
| **年5日取得チェック** | 取得不足者の管理者向けアラート (S-A-17)                                                                                            | **法令違反リスク回避**                                            |

### 2.2 含まない機能（明示）

| 機能                              | 含まない理由                                           | 代替手段                  |
| --------------------------------- | ------------------------------------------------------ | ------------------------- |
| 打刻修正申請ワークフロー          | 申請フローの設計が複雑、MVP では管理者直接修正で代替可 | 管理者が画面から直接修正  |
| 希望休申請ワークフロー            | 紙 / LINE で当面運用可                                 | 紙・LINE                  |
| 給与ソフト連携 CSV                | フォーマット要確認、画面の数値を見て手入力可           | 画面確認 + 手入力         |
| 月次集計のファイル出力            | 画面で見えれば月締めは可能                             | 画面表示                  |
| 拠点責任者ロール                  | 55名規模なら管理者1ロールで運用可能                    | 管理者で代行              |
| IC カード / 生体認証              | ハードウェア追加投資が必要                             | 共有タブレット + 暗証番号 |
| メール / LINE 通知                | 運用後にニーズを見極めたい                             | 口頭・LINE                |
| 位置情報チェック                  | プライバシー懸念、運用合意が必要                       | 共有タブレット利用        |
| 雇用契約書 / 労働条件通知書       | 既存運用が確立されている                               | 既存運用継続              |
| 社労士確認用 PDF                  | 必要書類を社労士と要確認                               | 既存運用継続              |
| キャリアアップ助成金 対応情報集計 | 仕様確定に時間が必要                                   | 別途 Excel 集計           |

## 3. Phase 2 — 「試験運用で必要性が確認できたら」

試験運用（5拠点 55名で1〜2か月）の結果を踏まえて優先順位を決める。

| 機能                               | 含む理由                 | 検討事項           |
| ---------------------------------- | ------------------------ | ------------------ |
| 打刻修正申請 (S-E-07, S-A-12)      | 紙申請を完全廃止するため | 承認フローの段階数 |
| 希望休申請 (S-E-08, S-A-13)        | 勤務表作成の手戻り削減   | 締切日設定の要否   |
| 申請ステータス画面 (S-E-09)        | 申請の見える化           | 通知との連動       |
| 月次集計 CSV / Excel 出力 (S-A-14) | 社労士・経理連携         | フォーマット標準化 |
| 給与ソフト連携 CSV                 | 給与計算工数削減         | 連携先ソフトの選定 |
| 拠点責任者ロール                   | 拠点長への権限委譲       | 権限粒度の合意     |
| 位置情報チェック（個人スマホ）     | 不正打刻防止             | プライバシー合意   |
| 通知（メール / LINE）              | 打刻忘れ・申請通知       | 配信手段の選定     |

## 5. スコープ変更ルール

- MVP の **削減** は妥当な理由があれば随時行う（運用シミュレーションで不要と判明した場合など）。
- MVP の **追加** は原則行わない。試験運用後に必要性が確認された機能は Phase 2 として整理する。
- Phase 2 / Future の **入れ替え** は要件オーナーの判断で随時行う。本ドキュメントを更新すること。
