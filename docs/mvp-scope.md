# MVP スコープ詳細

[`requirements.md`](requirements.md) の「機能要件」をフェーズ別に詳細化したもの。「含む / 含まない / 理由」を明示し、判断のブレを防ぐ。

> **2026-05-20 方針変更**: MVP を「人事 / 契約 / 書類 / 退職 / シフト制約 / 自動シフト」に絞り直した。打刻・勤怠承認・有給管理は Phase 2 / 3 に移動。

---

## 1. スコープ判定の方針

MVP 含めるか否かは以下の基準で判断する。

| 基準                           | 内容                                                                                                |
| ------------------------------ | --------------------------------------------------------------------------------------------------- |
| **管理者だけで完結できるか**   | 打刻のように現場合意・ハードウェア整備が要る機能は後回し。管理者画面で閉じる機能は MVP に含む       |
| **社内検証で価値が見えるか**   | 人事登録、契約、書類整備、シフト自動作成は社内で完結する → MVP                                      |
| **法令準拠**                   | 労働時間集計・有給付与・年 5 日取得チェックは法令準拠だが、打刻実績がないと意味がない → Phase 2 / 3 |
| **代替手段の有無**             | 紙・Excel で当面回せる機能は Phase 2 以降に回す                                                     |
| **個人情報セキュリティの要件** | 書類管理は PII を扱うため、MVP の段階で「セキュリティ万全」を前提に設計する                         |
| **UI 複雑度**                  | 高度な UI が必要で MVP の検証を遅らせる機能は Phase 2 以降に回す                                    |

## 2. MVP（Phase 1）— 「管理者画面だけで人事 + シフト作成が回る」

### 2.1 含む機能

| カテゴリ           | 機能                                                                                                            | 必須理由                                                           |
| ------------------ | --------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| 認証（完了）       | 管理者ログイン / 従業員ログイン / 共有タブレット打刻専用モード（導線のみ、打刻実装は Phase 2）                  | システム利用の前提                                                 |
| 拠点管理（完了）   | 拠点の追加・編集                                                                                                | 5 拠点運用の前提                                                   |
| 社員情報           | 基本情報・緊急連絡先・職種・雇用形態・在籍状況（active / on_leave / retired）・資格情報                         | 人事管理の最低限                                                   |
| 雇用契約           | 契約期間・更新管理・保険加入・退職金フラグ・キャリアアップ助成金フラグ（社労士確認用）                          | 契約履歴の整備、退職金通算判定の前提                               |
| 書類管理           | 履歴書・資格証明書・個人情報保護同意書・契約書 PDF・労働条件通知書 PDF・その他。署名 URL ダウンロード、監査ログ | PII の安全保管、社労士対応の前提                                   |
| 研修管理           | 研修名・種別・費用・日付の管理                                                                                  | キャリアアップ助成金 / 処遇改善加算の前提情報                      |
| 退職処理           | 退職ボタン（`employment_status=retired` に遷移）、退職日・理由、退職者リスト                                    | 在籍履歴の保持、データの物理削除を防ぐ                             |
| シフトパターン     | 早 / 日勤 / 遅 / 夜入 / 夜明 / 半日 / 拠点固有 / 休み系 の定義                                                  | 勤務表作成の前提。詳細は [`shift-patterns.md`](shift-patterns.md)  |
| 拠点シフト枠       | 拠点ごとの「平日 / 土 / 日祝 × シフトパターン」必要人員数                                                       | シフト自動作成の前提                                               |
| シフト制約         | 個人別の月間上限・夜勤上限・年収上限・不可曜日・月間出勤目標                                                    | パート 130 万円アラート、夜勤公平性の前提                          |
| シフト希望         | 希望休 / 希望夜勤 / 不可日 の登録（本人 + 管理者）                                                              | 自動配置で本人意向を反映するための入力                             |
| 月次シフト自動作成 | 制約付き greedy で 1 か月配置 → 警告 → 管理者調整 → 確定                                                        | 介護現場の運用必須機能。手動だけだと拠点 × 30 人分の作成負荷が重い |
| 月次勤務表編集     | 自動配置の手動微調整 (S-A-08)                                                                                   | 自動結果は提案、最終調整は管理者                                   |

### 2.2 含まない機能（明示）

| 機能                                        | 含まない理由                                                                  | 代替手段                                    |
| ------------------------------------------- | ----------------------------------------------------------------------------- | ------------------------------------------- |
| 打刻（出勤 / 退勤 / 休憩）                  | 共有タブレットの設置・現場合意が前提。ハード整備後に着手 (Phase 2)            | 紙のタイムカードを継続                      |
| 月別勤怠閲覧 / 集計                         | 打刻実績がないと表示するデータがない (Phase 2)                                | 紙集計                                      |
| 勤怠承認                                    | 打刻実装後に着手 (Phase 2)                                                    | 紙の押印フロー                              |
| 有給管理（自動付与・残数表示）              | 打刻 + 勤怠承認の後でないと実消化が記録できない (Phase 3)                     | Excel 管理                                  |
| 年 5 日取得チェック                         | 有給管理の後 (Phase 3)                                                        | 社労士確認                                  |
| 打刻修正申請ワークフロー                    | 打刻 + 承認フローの後 (Phase 3)                                               | 管理者が画面から直接修正                    |
| 希望休申請の承認フロー                      | MVP は `shift_preferences` で簡易運用、本格承認は Phase 3                     | MVP の希望入力 + 管理者確認                 |
| 給与ソフト連携 CSV                          | 打刻データがないと出力する数値がない (Phase 3)                                | 画面確認 + 手入力                           |
| 拠点責任者ロール                            | 55 名規模なら管理者 1 ロールで運用可能                                        | 管理者で代行                                |
| IC カード / 生体認証                        | ハードウェア追加投資が必要 (Future)                                           | Phase 2 で共有タブレット + 暗証番号         |
| メール / LINE 通知                          | 運用後にニーズを見極めたい                                                    | 口頭・LINE                                  |
| 位置情報チェック                            | プライバシー懸念、運用合意が必要                                              | 共有タブレット利用                          |
| 雇用契約書 / 労働条件通知書 の PDF 自動生成 | 既存運用が確立されている。MVP は **書類の保管** までを対象とし、生成は Future | 既存運用継続（Word テンプレ + 手動 PDF 化） |
| 社労士確認用 PDF（月次帳票）                | 必要書類を社労士と要確認 (Future)                                             | 既存運用継続                                |
| キャリアアップ助成金 対応情報集計           | MVP は **契約上のフラグと社労士確認用メモ** までを対象とし、申請支援は Future | 別途 Excel 集計                             |

## 3. Phase 2 — 「打刻運用が始められる状態」

社員管理基盤と自動シフトが完成し、現場で共有タブレットが整備できた段階で着手する。

| 機能                               | 含む理由           | 検討事項                               |
| ---------------------------------- | ------------------ | -------------------------------------- |
| 打刻（S-T-04 / S-E-01 へのボタン） | 全機能の基礎データ | タブレット選定、設置場所、ネットワーク |
| 月別勤怠閲覧（S-E-02 / S-E-03）    | 透明性・本人確認   | 夜勤跨ぎ表示                           |
| 勤怠承認（S-A-09）                 | 給与連携の前提     | 個別 / 一括承認のショートカット        |
| 月別勤怠集計（S-A-10）             | 月締めに必須       | 実労働 / 残業 / 深夜 / 休日            |

## 4. Phase 3 — 「試験運用で必要性が確認できたら」

試験運用の結果を踏まえて優先順位を決める。

| 機能                                          | 含む理由                 | 検討事項           |
| --------------------------------------------- | ------------------------ | ------------------ |
| 有給管理（自動付与、残数表示、有効期限）      | 法令準拠                 | 比例付与テーブル   |
| 年 5 日取得義務チェック (S-A-17)              | 法令違反リスク回避       | 警告強度のしきい値 |
| 打刻修正申請 (S-E-07, S-A-12)                 | 紙申請を完全廃止するため | 承認フローの段階数 |
| 希望休申請の本格ワークフロー (S-E-08, S-A-13) | 勤務表作成の手戻り削減   | 締切日設定の要否   |
| 申請ステータス画面 (S-E-09)                   | 申請の見える化           | 通知との連動       |
| 月次集計 CSV / Excel 出力 (S-A-14)            | 社労士・経理連携         | フォーマット標準化 |
| 給与ソフト連携 CSV                            | 給与計算工数削減         | 連携先ソフトの選定 |
| 拠点責任者ロール                              | 拠点長への権限委譲       | 権限粒度の合意     |
| 位置情報チェック（個人スマホ）                | 不正打刻防止             | プライバシー合意   |
| 通知（メール / LINE）                         | 打刻忘れ・申請通知       | 配信手段の選定     |

## 6. スコープ変更ルール

- MVP の **削減** は妥当な理由があれば随時行う（運用シミュレーションで不要と判明した場合など）。
- MVP の **追加** は原則行わない。試験運用後に必要性が確認された機能は Phase 2 / 3 として整理する。
- Phase 2 / 3 / Future の **入れ替え** は要件オーナーの判断で随時行う。本ドキュメントを更新すること。
