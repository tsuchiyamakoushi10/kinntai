# 手直しフィードバック設計 — 「公開までの手直しを減らす」

> 自動作成した勤務表を、管理者が手で直して公開するまでの **手直し作業を減らす** ための仕組みの設計メモ。
> あわせて、自動作成の出力を **管理者が手で作る理想形に近づける** ことを狙う。
>
> **AI（機械学習）は使わない。** 手直しの差分を集計し、しきい値を超えた「構造的な手直し」を、
> 既存の設定値（ツマミ）への変更として **管理者承認のうえで反映する** 半自動の仕組みとする。
> 自動作成本体は [auto-shift-design-v2.md](auto-shift-design-v2.md) を参照。ロジックは純粋関数 + テスト必須（CLAUDE.md §3.2）。

---

## 1. 背景と狙い

### 1.1 North Star

現在の運用は「自動作成（8割）→ 管理者が手直し → 公開」。価値の中心は **手直しの手間を減らすこと**。

- 手直しの量 ＝ 「自動作成の出力」と「管理者の理想形」の **ズレ** そのもの。
- このズレを小さくすれば、公開までの作業が減り、出力は理想に近づく。
- **同じ差分データが「成果の物差し」と「改善のタネ」を兼ねる**（§3）。「測る → 直す → また測る」の自己改善ループを、AIなしで回す。

### 1.2 やること / やらないこと

|          | 内容                                                                                           |
| -------- | ---------------------------------------------------------------------------------------------- |
| やる     | 手直し差分を貯める / 手直し件数を見える化 / 構造的な傾向を既存ツマミへの変更として提案         |
| やらない | 機械学習モデルの学習・推論。ブラックボックスな自動変化。労務ルール（有給・法定労働時間）の学習 |

「説明できるフェーズ式」という v2 の設計思想（なぜこの配置か言える）を壊さないため、学習ではなく **見えるルールへの変換** に徹する。

## 2. 全体像（サイクル）

```
自動作成(提案)  →  管理者が手直し  →  公開(ShiftPublication)
     │                  │                    │
     │              ← 差分(手直し) を記録 →   │
     │                                        │
     └───────  傾向を集計 → 承認 → ツマミへ反映 ──┘  (翌月の提案が理想に近づく)
```

- 「提案」＝ `ShiftGenerationRun`（`status=DRAFT`）が作った自動配置。
- 「公開」＝ `ShiftPublication` に行ができた状態（提案の確定 `CONFIRMED` とは別概念。手動作成拠点には Run が無い）。
- 学習・提案の入力は **提案 ↔ 公開時点の最終形の差分**。窓は `generatedAt` 〜 `publishedAt`。

---

## 3. KPI 定義

### 3.1 手直し件数

- **主指標**：拠点 × 対象月ごとの **手直しセル数**（提案と最終形で内容が変わったセルの数）。
- 補助：**手直し率** ＝ 手直しセル数 ÷ 営業セル数（配置対象の（従業員×営業日）数）。拠点規模の違いを吸収する。
- 目標：主指標を **右肩下がり** にする。

### 3.2 手直しは 2 種類。ゼロは目指さない

| 種類       | 例                                              | 扱い                                  |
| ---------- | ----------------------------------------------- | ------------------------------------- |
| **構造的** | 毎月同じ曜日を公休に直す / いつも同じ記号へ置換 | ルール化して潰す対象。**これを 0 へ** |
| **突発**   | 急な休み・その月だけの事情                      | そもそも学習しない。残ってよい        |

到達目標は「構造的な手直しが消え、残るのは真の突発だけ」。突発を構造と誤認しないための **しきい値**（§5.3）が肝。

## 4. 差分データの持ち方

### 4.1 課題：手直しすると元の提案値が消える

現状、手直しは同じ `Shift` 行を上書きする（`updatedBy` が変わる）。手直しされたセルの **元の自動提案値は残らない**。差分を正確に取るには、提案時点のスナップショットが必要。

- 手直しセルの識別自体は既存の仕組みで可能：「自動由来かつ未編集（`generationRunId===run.id && updatedBy===generatedById`）」でないセル ＝ 手直し（[auto/actions.ts](../src/app/admin/shifts/auto/actions.ts) の `loadProtectedCells`）。
- ただし「**何を** 何に変えたか」までは、提案スナップショットが無いと復元できない。

### 4.2 方針：提案スナップショットを保存（推奨）

`saveDraftRun` 時に、その回の **純粋な自動提案** を不変（immutable）で保存する新テーブルを追加する。

```
model ShiftGenerationProposal {   // 提案の写し。手直しでは触らない。
  id                   uuid  PK
  generationRunId      uuid  FK → ShiftGenerationRun
  employeeId           uuid
  workDate             date
  proposedShiftPatternId uuid?   // 提案した記号（公休含む。無配置は null）
  @@unique([generationRunId, employeeId, workDate])
}
```

- `ShiftGenerationRun.algorithmVersion` を必ず記録し、**エンジン変更をまたいだ差分を区別**できるようにする（既に列あり）。
- 差分 ＝ 「公開時点の `Shift`」 vs 「同月・同拠点の `proposal`」を（employee × workDate）で突き合わせ。

> 代替案：生成は決定論なので「当時の入力から提案を再計算」して突き合わせる手もある。ただし配置基準・従業員・制約が後から変わると再現がずれるため、**スナップショット方式を推奨**（確実・軽い）。

### 4.3 差分の記録（任意・第2段）

集計を軽くするため、公開時に確定差分を要約テーブルへ書き出してもよい（`ShiftCorrection`：run, employee, date, from_symbol, to_symbol, kind）。MVP では §4.2 のスナップショットと最終 `Shift` の突き合わせクエリで足りる。

## 5. 手直しの分類と集計

### 5.1 手直しの種別（単純ルールで分類）

| 種別         | 判定                           |
| ------------ | ------------------------------ |
| 休 → 勤      | 提案=公休/有休 → 最終=勤務     |
| 勤 → 休      | 提案=勤務 → 最終=公休/有休     |
| 記号差し替え | 同一人・同一日で 勤務A → 勤務B |
| 夜勤の増減   | 夜入/夜明 コマの増減           |

### 5.2 集計軸

- （従業員 × 曜日）… 「A さんは金曜がいつも休みに直る」
- （従業員 × 記号ペア）… 「A さんはショ短A → 半日A に毎回直る」
- （拠点 × 日種）… 「土曜がいつも不足（赤）で手足しされる」
- （従業員 × 月次の勤務日数差）… 「目標日数を毎月 手で調整している」

### 5.3 しきい値（突発を除外）

- 例：**直近 3 か月で 2 回以上**、または **3 か月連続** 同じパターン → ルール化候補。
- 1 回きりは無視（突発とみなす）。
- しきい値は定数で外出しし、運用で調整。

---

## 6. 傾向 → 既存ツマミへの変換

見つけた構造的な手直しは、**既存の設定値** に落とし込む（新しい概念を増やさない。CLAUDE.md §7 の「値は外出し」方針と一致）。

| 観測パターン                 | 変換先（既存の値・仕組み）                 | 反映        |
| ---------------------------- | ------------------------------------------ | ----------- |
| 毎月同じ曜日を公休に直す     | 個人の定例希望休 / ソフトな不可曜日        | 提案 → 承認 |
| 特定の曜日区分が恒常的に不足 | `office_coverage_demands`（配置基準）      | 提案 → 承認 |
| ある人の記号を毎回置換       | その人の常用記号 / roster の `fixedSymbol` | 提案 → 承認 |
| 夜勤を毎回 減らす/増やす     | `maxNightShiftsPerMonth` / 夜勤可否        | 提案 → 承認 |
| 目標日数を常時 調整          | `targetMonthlyWorkDays`（個別上書き）      | 提案 → 承認 |

---

## 7. 反映は管理者承認（human-in-the-loop）

- 自動で設定を書き換えない。**「こういう手直しが続いています。ルール化しますか？」** と管理者に提案し、承認で初めて反映。
- 提案には必ず **根拠** を添える（例：「直近 3 か月で金曜公休の手直しが 3 回」）。説明可能性を担保。
- 却下・保留も記録し、同じ提案を繰り返し出さない。

---

## 8. AI を使わない理由 / 将来の余地

- この規模（5 拠点・約 55 名・月次）では学習データが小さく、機械学習は過学習しやすくノイズを拾う。
- やりたいことは `GROUP BY` と件数しきい値で表現でき、統計モデルすら不要。
- **説明できる**・**少データに強い**・**管理者が制御できる**・**実装が軽い** の 4 点で、AI なしが優る。
- 将来 AI を検討しうるのは「自由記述メモから意図を汲む」「多数の制約が複雑に絡む最適化を一括で解く」等に踏み込むとき。まず集計で回し、物足りなければ、で十分。

---

## 9. 注意点

- **少データ**：しきい値必須。1 回の例外を学習しない（§5.3）。
- **労務ルールは学習させない**：有給日数・法定労働時間などはルール固定（CLAUDE.md §5）。
- **説明可能性**：隠れた重み付けにせず、必ず見えるルールへ変換。
- **個人情報**：手直し傾向も PII を含む。ログ・エラートラッキングに氏名等を出さない（CLAUDE.md §5）。

## 10. 段階導入

| 段階         | 内容                                                                   | 価値 / リスク                                        |
| ------------ | ---------------------------------------------------------------------- | ---------------------------------------------------- |
| **①（MVP）** | 提案スナップショット保存（§4.2）＋「今月の手直し件数」を管理画面に表示 | 効果を数値で見える化・改善の土台。リスク低           |
| **②**        | 手直しの分類・集計（§5）と一覧（「今月 手直しされた点」）              | どこに手間がかかるか特定。学習はまだしない           |
| **③**        | 傾向集計 → ツマミ変更の提案（§6・§7）。承認で反映                      | 構造的な手直しを潰す。運用データが数か月たまってから |

① だけでもリスクが低く価値が高い（作業のボトルネックが見え、改善が証明できる）。
