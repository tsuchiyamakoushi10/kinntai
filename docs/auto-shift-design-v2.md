# 月次シフト自動作成 設計 v2 — フェーズ式 + 拠点別設定

> v1 ([auto-shift-design.md](auto-shift-design.md)) の「スコア合算 greedy」を、**説明できる優先順位(フェーズ式)** に作り替える再設計。あわせて、これまでハードコードしていた調整値を **拠点別・スタッフ別に外出し** する。
>
> 方針・主要論点はオーナー確認済み (2026-06-08、§8 設計判断ログ)。実装は §8.1 の DB 追加([database-design.md](database-design.md) / [employee-master.md](employee-master.md) への追記)から着手する。DB I/O はサーバアクション側、ロジックは純粋関数 + テスト必須 (CLAUDE.md §3.2)。

---

## 1. 背景と狙い

### 1.1 なぜ作り替えるか

v1 は「(従業員 × 日 × パターン)」ごとに点数を合算し最大の人を採る方式。動くが、

- **なぜこの人がここに入ったのか説明しづらい**(重みの足し算の結果)。
- 連勤上限・夜勤上限・年収上限・スコアの重みが **全拠点共通のハードコード**。「デイは連勤◯日まで」等の拠点差を出せない。
- 梨花だけ別アルゴリズム ([rika/generate.ts](../src/lib/shift/rika/generate.ts)) で、将来統一しづらい。

### 1.2 v2 の考え方

シフトを組む人間の手順そのままに、**優先順位の決まったフェーズ**で順に埋める。これは梨花でやっている Tier1(正社員先)→ Tier2(パートで穴埋め)を全拠点へ一般化したもの。完璧な最適解は狙わず、**「8割組む → 過不足を色で見る → 人が直す」**(v1 と同じ哲学) を維持する。

---

## 2. 決定の優先順位(フェーズ)

```
0. 前処理        … 日付・配置基準・各人の「入れない日」を確定
1. 需要を確定    … 営業日 × 配置基準(シフト枠) で「各日・各シフト 何人必要か」
2. 夜勤を先取り  … 夜勤枠を「夜勤希望が残る人」優先で確保 (月の夜勤上限内)   ★案A
3. 正社員を配置  … 残る勤務枠を、勤務日数(目標)に届くよう正社員で埋める
4. パートで穴埋め … 残った枠をパートで。希望休と年収上限を考慮
5. 公休埋め      … どの枠にも入らない日は公休
```

**全フェーズ共通の絶対ルール(ハード制約)**

- 希望休 / 勤務不可 / 不可曜日 / 雇用期間外の日には**入れない**(正社員・パート問わず)。
- 連勤上限(拠点設定。既定 6 日)を超えない。
- 1 人 1 日 1 コマ(同日重複なし)。
- 夜勤を入れたら翌日に必ず夜明けを対で置く。

> パートの希望休も②④の両方で必ず除外する(オーナー指摘 2026-06-08)。

## 3. 各フェーズ詳細

### Phase 0 — 前処理

- 当月の日付一覧 + 各日の `day_kind`(平日 / 土 / 日祝)。
- 各従業員の**不可日集合**:不可曜日、希望休 / 勤務不可の申請日、既存占有、入社前・退職後。
- 前月末 NIGHT_IN → 当月 1 日 NIGHT_OUT を最優先で確保。
- 各従業員の**当月の状態**を初期化:目標勤務日数、夜勤希望日数、夜勤上限、年収上限、累積労働(年収見込み用)。

### Phase 1 — 需要を確定(営業日 & 配置基準)

- シフト枠(`office_shift_quotas`:拠点 × 日種 × パターン × 必要人数)を当月の各日に展開し、**slot**(date × pattern × 必要数)を作る。
- **営業日**は「その日の日種に必要数 > 0 の枠がある日」。日祝の枠が 0 ならその日は配置しない(= 実質休業)。
- 夜勤枠(NIGHT_IN)もここに含める。

### Phase 2 — 夜勤を先取り(★案A:独立フェーズ)

夜勤枠を**日勤より先に**埋める。介護現場では夜勤を先に固めるのが実務に近いため。

- 候補:その日に入れる人(不可日でない / 未割当 / 翌日も空いていて夜明けが置ける / 月の夜勤上限内)。
- **優先順**:
  1. **夜勤希望日数がまだ残っている人**(`希望 − 当月実績` が大きい人ほど優先)。
  2. 当月の夜勤回数が少ない人(偏り防止)。
  3. 同点は seed 乱数で安定化(決定論)。
- 月の夜勤上限(拠点設定。既定 5。個人で上書き可)を超えない。人員が足りずやむを得ず超える場合のみ配置し `NIGHT_SHIFT_OVER_LIMIT` 警告(v1 論点 D を踏襲)。
- 配置した夜勤は本人の勤務日数にカウント(Phase 3 の目標計算に反映)。

### Phase 3 — 正社員を配置

残る勤務枠を**正社員(+契約社員)**で、各自の**勤務日数(目標)に届くよう**埋める。

- slot を 日付昇順 → パターン順 に走査。
- 候補:正社員で、その日入れて(不可日でない / 未割当 / 連勤上限内)。
- **優先順**:
  1. **目標までの残り日数が多い人**(契約日数に皆が近づくよう公平に)。
  2. 当月の勤務日数が少ない人。
  3. 同点は seed 乱数。
- 目標に達した正社員は**それ以上は優先しない**(超過配置は避ける)。ただし枠が埋まらず必要な場合は配置可(過不足は警告で可視化)。
- 月末に目標へ届かなかった正社員は `TARGET_WORKDAYS_UNREACHED` 警告。

### Phase 4 — パートで穴埋め

残った枠を**パート**で埋める。

- 候補:パートで、その日入れて(**希望休でない** / 不可日でない / 未割当 / 連勤上限内)、**年収上限を超えない**人。
- **優先順**:
  1. 当月の勤務日数が少ない人(公平分散)。
  2. 年収に余裕がある人。
  3. 同点は seed 乱数。
- 年収見込みが上限(拠点設定。既定 130 万。個人で上書き可)を超えないパートを優先する。上限を超えてしまう配置は避けるが、他に埋められる人がいない場合のみ最終手段で配置し `INCOME_CAP_EXCEEDED` 警告を出す(人員確保 > 年収厳守。夜勤上限の override と同じ考え方)。年収見込みは [`income-projection.ts`](../src/lib/shift/income-projection.ts) を再利用。

### Phase 5 — 公休埋め

どの枠にも入らなかった日を公休(OFF)で埋める。不可日(雇用期間外・既存占有)は埋めない。

## 4. 調整できる値(設定レイヤ)

これまでハードコードだった値を**拠点別 / スタッフ別**に外出しする。現状値を既定にすれば挙動は変えずに「触れるように」できる。

拠点別の値は**新テーブル `office_shift_setting`(office と 1:1)** に持たせる(2026-06-08 確定)。`office` 列追加で肥大化させず、行が無い拠点は既定値で扱う。

### 4.1 拠点別(office 単位)

| 値                                 | 既定                | 使うフェーズ           | 現状の置き場所                               |
| ---------------------------------- | ------------------- | ---------------------- | -------------------------------------------- |
| 配置基準(日種 × パターン × 必要数) | —                   | Phase 1                | ✅ 既に DB(`office_shift_quotas`)            |
| 連勤上限                           | 6 日                | 全フェーズ(ハード制約) | ❌ 定数 `MAX_CONSECUTIVE_WORK_DAYS`          |
| 月の夜勤上限(既定値)               | 5 回                | Phase 2                | ❌ 定数 `DEFAULT_MAX_NIGHT_SHIFTS_PER_MONTH` |
| パート年収上限(既定値)             | 1,300,000 円        | Phase 4                | ❌ placement.ts に直書き                     |
| 正社員 目標勤務日数の決め方        | 式(週日数 × 月週数) | Phase 3                | ❌ 式が定数                                  |

### 4.2 スタッフ別(employee 単位、既存の従業員制約を活用)

| 値                       | 使うフェーズ | 現状                        |
| ------------------------ | ------------ | --------------------------- |
| 希望休 / 勤務不可        | 全フェーズ   | ✅ シフト希望(preferences)  |
| 不可曜日                 | 全フェーズ   | ✅ 従業員制約               |
| 目標勤務日数(個別上書き) | Phase 3      | ✅ `targetMonthlyWorkDays`  |
| 月の夜勤上限(個別上書き) | Phase 2      | ✅ `maxNightShiftsPerMonth` |
| **夜勤希望日数**         | Phase 2      | ⚠️ 新規(下記)               |
| 年収上限(個別上書き)     | Phase 4      | ✅ `annualIncomeCapYen`     |

> **夜勤希望日数** は v2 の Phase 2 で中心になる新項目。「この人は月◯回まで夜勤に入りたい」を持たせ、その回数まで優先的に夜勤を割り当てる。**従業員登録(従業員マスター)に「月◯回」の項目を追加**して保持する(2026-06-08 確定)。現状の「夜勤希望(する/しない)」フラグはこの回数に置き換え/統合する。

### 4.3 内部のみ(現場には出さない)

- スコア / 優先度の細かい重み付けは v2 では**明示フェーズの並び順**に置き換わるため、ユーザー設定にはしない。微調整が要る場合はコード(定数)で行う。現場 UI に出すのは §4.1〜4.2 の業務的に意味がある値だけ(CLAUDE.md:専門用語を出さない)。

## 5. 警告コード(v1 から継続 + 追加)

| code                        | 内容                                               | 主因フェーズ |
| --------------------------- | -------------------------------------------------- | ------------ |
| `QUOTA_UNDERFILLED`         | 必要人数に届かない枠                               | 1〜4         |
| `NIGHT_SHIFT_OVER_LIMIT`    | 月の夜勤上限を超えて配置                           | 2            |
| `NIGHT_PREF_UNMET`(新)      | 夜勤希望日数に届かなかった                         | 2            |
| `TARGET_WORKDAYS_UNREACHED` | 正社員の目標勤務日数に未達                         | 3            |
| `INCOME_CAP_EXCEEDED`       | パートの見込み年収が上限超過                       | 4            |
| `PREV_MONTH_NIGHT_HANGING`  | 前月夜勤の夜明けを引き継げない                     | 0            |
| `REQUEST_OFF_UNHONORED`(新) | 希望休を満たせず勤務になった(原則出ない想定の保険) | 全           |

## 6. 実装方針

- [`placement.ts`](../src/lib/shift/auto-generator/placement.ts) を「スコア合算」から「フェーズ式」へ組み替える。`scoreCandidate` は各フェーズの**優先順比較関数**に分解。
- 入力 `GenerateInput` / 出力 `GenerateOutput` の形は極力維持(サーバアクション層・UI 影響を最小化)。
- 拠点別設定は新テーブル `office_shift_setting`(§8②)。まずは既定値を定数で持ち、`GenerateInput` で受け取れる形にしておき、後で DB 配線するだけにする。
- 決定論性(同じ入力 + seed → 同じ結果)は維持。
- 梨花は当面 [rika/generate.ts](../src/lib/shift/rika/generate.ts) のままだが、v2 が安定したらこのフェーズ式へ寄せて統一する。

## 7. テスト方針(必須・CLAUDE.md §3.2)

1. Phase 1:枠どおりの slot 数が出る / 日祝 0 枠の日は配置しない。
2. Phase 2:夜勤希望が残る人が優先される / 月上限を超えない / 翌日に夜明けが入る。
3. Phase 3:正社員が目標日数に向けて公平に配置される / 目標超過は避ける。
4. Phase 4:パートの希望休が除外される / 年収上限で打ち止め。
5. 全フェーズ:希望休・連勤上限が必ず守られる。
6. 埋まらない枠が `QUOTA_UNDERFILLED` で出る。
7. 同じ seed で再現する(決定論)。
8. 拠点設定(連勤上限・夜勤上限・年収上限)を変えると結果が変わる。

## 8. 設計判断ログ(2026-06-08 確定)

| ID  | 論点                               | 確定事項                                                                                                                      |
| --- | ---------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| ①   | **夜勤希望** の持ち方              | **「月◯回」を従業員登録(従業員マスター)に追加**。Phase 2 はこの回数まで優先配置。既存の「する/しない」フラグは回数へ統合。    |
| ②   | 拠点別設定の**保存先**             | **新テーブル `office_shift_setting`(office と 1:1)**。行が無ければ既定値。`office` への列追加はしない。                       |
| ③   | 正社員の目標日数は **厳守 / 目安** | **目安**。届かなければ `TARGET_WORKDAYS_UNREACHED` 警告のみ。最終的な過不足は管理者が手動調整する前提。                       |
| ④   | 「営業日」の表し方                 | **「その日種に必要数 > 0 の枠がある日 = 営業日」** とし、専用フラグは持たない。梨花のような固定営業日は専用ロジック側で保持。 |

### 8.1 実装で必要になる DB 追加(②①に伴う)

- `office_shift_setting`(新規):`office_id`(unique)、`max_consecutive_work_days`、`default_max_night_shifts_per_month`、`default_annual_income_cap_yen`、`fulltime_target_workdays_mode` 等。詳細は実装前に [database-design.md](database-design.md) へ追記。
- `employee`(または従業員制約):`desired_night_shifts_per_month`(月の夜勤希望回数)を追加。従業員マスター CSV ([employee-master.md](employee-master.md)) の列追加も検討。
