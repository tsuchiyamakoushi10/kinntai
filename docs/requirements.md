# 要件定義書

## 1. プロジェクトの目的

小規模〜中規模の介護事業所が、**勤怠管理・勤務表作成・有給管理・従業員情報管理** を一気通貫で行える Web アプリを提供する。

主なユーザーは管理者（事業所長・サービス提供責任者など）と現場の介護職員。専用ハードウェアを購入することなく、既存の **共有タブレット** と **個人スマホ** だけで運用できることを重視する。

## 2. 想定規模（初期）

| 項目                      | 想定値                                                                                                      |
| ------------------------- | ----------------------------------------------------------------------------------------------------------- |
| 拠点数                    | 5拠点（ナーシングホーム結いの心 / デイサービス結いの心 / ショートステイ結いの心 / デイサービス梨花 / 厨房） |
| 従業員数                  | 約55名（全拠点合計、正社員30 + パート25）                                                                   |
| 1拠点あたり同時打刻ピーク | 10〜15名程度（朝の出勤集中時、24h 施設の夜勤交代含む）                                                      |
| 管理者ユーザー数          | 数名〜10名以内                                                                                              |
| シフトパターン数          | 約30種（拠点専用パターン多数。詳細は [`docs/shift-patterns.md`](shift-patterns.md)）                        |

## 3. ユーザーロール

| ロール                               | 想定する人                                                      | 主な権限                                                     |
| ------------------------------------ | --------------------------------------------------------------- | ------------------------------------------------------------ |
| **管理者**                           | 事業所長、事業所責任者、サービス提供責任者、管理事務            | 全拠点の従業員管理、勤務表編集、勤怠承認、有給付与、各種設定 |
| **拠点責任者**（Phase 2 で分離検討） | 拠点リーダー                                                    | 自拠点のみの勤怠承認・勤務表編集                             |
| **従業員**                           | 正社員、契約社員、パート、夜勤スタッフ など、現場で勤務する全員 | 自分の勤怠閲覧、打刻、希望休 / 修正申請（Phase 2）           |

> **MVP では「管理者」と「従業員」の 2 ロールのみ**。「事業所責任者」は **管理者の一種** として扱う。拠点責任者ロールの分離は Phase 2 で再検討する。ロール別の権限詳細は [`user-roles.md`](user-roles.md) を参照。

## 4. 機能要件

> **2026-05-20 方針変更**: 打刻機能・勤怠管理・有給管理は Phase 2 以降に移動した。MVP では「人を入れる・契約を結ぶ・必要書類を揃える・退職を記録する」「シフト枠を制約付きで自動生成する」までを最優先で整える。打刻機能はハードウェア整備と現場合意を要するため、人事系を先行させて社内検証ができる状態を作る。

### 4.1 MVP（フェーズ1）

#### 認証（実装済）

- 管理者ログイン（メール + パスワード）
- 従業員ログイン（メール + パスワード、または 従業員コード + 暗証番号）
- 共有タブレット用「打刻専用モード」: ログイン状態を維持しつつ、本人を暗証番号で選択して打刻（打刻機能の実装は Phase 2 だが、認証導線は MVP で先行）

#### 拠点管理（実装済）

- 拠点の追加・編集
- 拠点ごとのタイムゾーン設定（既定 `Asia/Tokyo`）

#### 社員情報管理

人事担当が「採用 → 在籍 → 退職」を一気通貫で扱える形にする。

- 基本情報: 社員番号、氏名（漢字・カナ）、生年月日、性別、電話番号、住所、備考
- 緊急連絡先: 氏名、続柄、電話番号
- 雇用区分: 所属拠点、職種（介護職員 / 看護職員 / 生活相談員 / ケアマネ / 事務 など）、雇用形態（正社員 / 契約社員 / パート）
- 在籍状況: 在籍 / 休職 / 退職（`employment_status` enum）。`retired_at` だけでは休職を表現できないため別カラムで保持
- 資格情報: 介護福祉士、初任者研修、実務者研修などの保有資格と取得日・登録番号

#### 雇用契約管理

社労士確認に耐える契約履歴を持つ。1 従業員に複数契約を時系列で積む（更新ごとに新規レコード）。

- 契約期間: 契約開始日、契約終了日（正社員は NULL = 無期）
- 契約条件: 雇用形態（契約時点）、所定労働時間/日、所定労働日数/週、賃金形態（時給/月給）、賃金額
- 更新管理: 更新有無、更新回数、更新上限の有無、更新上限回数、更新判断基準
- 保険・福利厚生: 雇用保険加入有無、社会保険加入有無
- 退職金: 退職金対象有無（手動上書き可、後述の自動判定値をサジェスト表示）
- キャリアアップ助成金: 助成金対象として記録するフラグ + 社労士確認用メモ。**助成金申請そのものは扱わず、社労士確認用の情報整理として扱う**
- 紐づく書類: 雇用契約書 PDF、労働条件通知書 PDF を `employee_documents` から参照

> 退職金判定 (正社員として通算 3 年経過): `employment_contracts` のうち `employment_type=full_time` の通算期間を計算し、3 年以上で「対象」をサジェスト表示。最終判断は管理者が `retirement_allowance_eligible` フラグで確定する。

#### 書類管理

PII を含むため、保存・配信ともにセキュリティを優先する（詳細は [`docs/database-design.md §6`](database-design.md#6-設計上の留意点)）。

- 書類種別: 履歴書 / 資格証明書 / 個人情報保護同意書 / 雇用契約書 PDF / 労働条件通知書 PDF / その他
- 保存方針: ファイル本体は外部オブジェクトストレージ（本番は Supabase Storage または S3 互換）に保存し、DB には storage_key のみ。ローカル開発は `./storage/` (gitignore) で代替
- 配信方針: ダウンロードは短期署名 URL（5 分有効）。管理者ロールのみアクセス可
- 監査ログ: 誰がいつ何の書類を閲覧・ダウンロードしたかを `document_access_logs` に記録
- 期限管理: 資格証明書など `expires_on` を持つ書類は、期限切れ前に管理者ダッシュボードへ通知

#### 研修管理

キャリアアップ助成金対応・処遇改善加算対応の前提情報として整備する。

- 研修名、研修種別（有料 / 会社負担）、費用、研修日、備考
- 修了証ファイルがあれば `employee_documents` から紐付け

#### 退職処理・退職者リスト

- 社員詳細画面に「退職処理」ボタンを置き、削除ではなく `employment_status=retired` に状態遷移
- 入力項目: 退職日、退職理由（テキスト）
- 退職者リスト画面: 退職日・退職理由・最終所属拠点でフィルタ可能
- 退職者の勤怠・契約・書類データは全て保持（参照のみ可能）

#### シフト制約条件管理

シフト自動作成と労務トラブル防止の両方の前提として、個人単位の制約を蓄積する。

- 月間勤務時間上限（任意）
- 1 日の勤務時間上限（任意）
- 月間夜勤上限（正社員既定 5 回、人員不足時の超過許可フラグ）
- **パート 年収 130 万円アラート**: 個人別 `annual_income_cap_yen` を設定。割当シフトの労働時間 × 契約時給から「見込み年収」を算出し、上限の 80% / 100% で段階アラート
- 勤務できない曜日（複数指定可）
- 正社員の月間出勤目標日数（既定 21 日）
- シフト希望: 希望休 / 希望夜勤 / 勤務不可日 を月別カレンダーで本人 or 管理者が登録

#### 1 か月シフト自動作成

事業所の人員配置要件を満たすシフトを 1 か月分自動配置する。

- **拠点別 シフト枠定義**: 拠点ごとに「平日 / 土 / 日祝 × シフトパターン（早・日勤・遅・夜入 など）」の必要人数を設定（`office_shift_quotas`）
- 自動配置の優先順位:
  1. 希望休 / 勤務不可日 / 不可曜日 を **絶対制約** として除外
  2. 正社員に対し月間出勤目標日数（既定 21 日）を満たす配置を優先
  3. 夜勤は「希望夜勤」を持つ従業員から先に割当。本人月間上限を超えない範囲で配置
  4. 5 回上限を超える夜勤割当は最終手段（人員不足警告つきで明示）
  5. 残り枠にパートを当てる。労働時間上限・年収見込み上限を侵さない範囲で
- 結果は `shift_generation_runs` に履歴として残し、`shifts` に下書き状態で書き出す
- **管理者が最終調整**: 自動配置はあくまで提案。管理者が手動で動かして確定する
- 自動アルゴリズムは初期 greedy 実装。配置ロジックは `src/lib/shift/auto-generator/` に隔離してテストする

#### シフトパターン設定

- 名称、開始時刻、終了時刻、休憩時間、種別（早 / 日勤 / 遅 / 夜入 / 夜明 / 半日勤務 / 拠点固有パターン / 休み系）
- カラー設定（勤務表上の視認性のため）
- 拠点固有 / 全拠点共通の切り替え（`shift_patterns.office_id` で表現）
- 複合パターン（午前有休 + 午後勤務など）に対応する `paid_leave_units` 値の設定（詳細は [`docs/shift-patterns.md`](shift-patterns.md)）

### 4.2 Phase 2（試験運用前）— 打刻・勤怠

社員管理基盤と自動シフトが完成し、現場でハードウェア（共有タブレット）が整備できた段階で着手する。

#### 打刻

- 出勤打刻 / 退勤打刻 / 休憩開始 / 休憩終了
- 打刻時に拠点を自動判定（タブレットは拠点固定、スマホは所属拠点をデフォルト）
- 打刻履歴の即時表示（直近 5 件程度）

#### 勤怠閲覧 / 承認 / 集計

- 従業員: 自分の月別勤怠（出勤・退勤・休憩・実労働時間・残業）一覧
- 管理者: 拠点 / 個人別の月別勤怠一覧、未承認打刻のハイライト
- 月締めの個別 / 一括承認
- 月別集計（出勤日数、実労働、所定内/外、深夜、休日、有休消化）— 画面表示

### 4.3 Phase 3（試験運用後）— 有給・申請系

- 有給管理（雇い入れ日基準の自動付与、比例付与、繰越、2 年失効、残数表示）
- **年 5 日取得義務チェック**: 年 10 日以上付与された従業員の取得不足アラート
- 打刻修正申請（従業員 → 管理者）と承認フロー
- 希望休申請の正式ワークフロー化（MVP の `shift_preferences` 入力 + 承認フローを追加）
- 月次集計の CSV / Excel 出力
- 給与計算ソフト向け CSV エクスポート
- 拠点管理者ロールの分離と権限制御
- スマホでの打刻時の位置情報チェック（拠点近傍判定）
- 通知（打刻忘れ、申請通知）— メールまたは LINE

### 4.4 Future（中長期）

- IC カード打刻 / 生体認証打刻
- 雇用契約書 PDF の自動生成・電子署名
- 労働条件通知書 PDF の自動生成
- 社労士確認用 PDF 出力（月次勤怠・有給・労務情報をまとめた帳票）
- キャリアアップ助成金 申請に必要な情報集計・出力
- 処遇改善加算 / 特定処遇改善加算 の算定支援
- マルチテナント化（複数法人での利用）
- 介護記録 / 訪問記録との連携

## 5. 非機能要件

| カテゴリ         | 要件                                                                    |
| ---------------- | ----------------------------------------------------------------------- |
| **同時利用**     | 同時打刻 30 req/s 程度を許容（朝のピーク × 安全係数）                   |
| **レスポンス**   | 打刻アクションは応答 1 秒以内を目標                                     |
| **稼働率**       | 99% 程度（小規模事業所向けのため、夜間メンテ可）                        |
| **データ保持**   | 勤怠データは最低 5 年保持（労基法 3 年 + 余裕）                         |
| **バックアップ** | 日次フルバックアップ、最低 30 日分保持                                  |
| **タイムゾーン** | サーバ・DB ともに UTC で保存、表示は `Asia/Tokyo`                       |
| **ブラウザ対応** | 最新版の Chrome / Safari / Edge。iPad Safari 必須                       |
| **画面サイズ**   | 共有タブレット（1024×768 横向き）と スマホ（375×667 縦向き）両対応      |
| **セキュリティ** | パスワード bcrypt ハッシュ、HTTPS 必須、CSRF 対策、ログイン試行回数制限 |
| **個人情報保護** | PII はログに出さない、エクスポート時に管理者ログを残す                  |

## 6. 法令・制度の前提

- **労働基準法**: 労働時間（1日8時間 / 週40時間）、割増賃金（深夜 25%、休日 35%、時間外 25%）、年次有給休暇の付与基準
- **介護保険法 / 介護報酬**: 直接の算定はしないが、将来の処遇改善加算対応のための情報項目を確保する
- **個人情報保護法**: 従業員情報の安全管理措置

法令準拠が関わるロジック（特に有給付与日数、深夜割増、夜勤跨ぎ判定）は、根拠とともにコード内コメントまたはドキュメント側に記録する。

## 7. スコープ外（明示）

以下は本アプリでは扱わない。

- 給与計算そのもの（時間集計までを行い、給与ソフトに連携する想定）
- 高度な AI 配置最適化（MVP のシフト自動作成は制約付き greedy ベース。OR-Tools などの最適化ソルバーは扱わない）
- 利用者（介護を受ける方）の情報管理
- 介護記録・サービス提供記録
- 請求業務（国保連連携など）
- キャリアアップ助成金の申請業務そのもの（社労士確認用の情報整理までを担う）
